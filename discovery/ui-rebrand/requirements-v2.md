## 1. なぜ v2 か — 改訂背景

### v1 の問題点

ユーザーインタビュー（Q1〜Q10）を経て、v1 AI ブレインタブの以下の問題が明らかになった。

| 問題 | 詳細 |
|---|---|
| グラフが意味不明 | ノードと線が並んでいても「何がわかるのか」が伝わらない |
| 数字だけで温度がない | 記事数・知識数を見ても「で、何？」という感覚 |
| 構造が分散している | 縦・横・グラフと視線が定まらない |
| 演出が多すぎる | パルス / フェードイン / 横スクロールが「情報密度の薄さ」を隠していた |

### v2 のコアコンセプト変更

| | v1 | v2 |
|---|---|---|
| トーン | 「AI が育っている」演出・ゲーム感 | 「客観データを静かに見せる」ダッシュボード |
| 主役 | グラフ / アニメーション | タグ別カテゴリ＋記事数 |
| 情報構造 | 縦・横・グラフで分散 | 縦スクロール 1 本のみ |
| グラフ | あり（ナレッジマップ） | **廃止** |
| 横スクロール | あり（最近の動きカード） | **廃止** |

> **v2 コアコンセプト:**  
> **「自分がどの分野の知識を積み上げているか、客観的なデータとして静かに見せる」**

## 2. アプリコンセプト（v1 継承）

> **「Safari で読んだ記事が自動で吸収され、あなた専用の iPhone AI がどんどん育っていく」**

ユーザーは意識せず知識を蓄積し、ふと画面を開くと自分の AI が成長している。  
その成長を確認する快感がモチベーションの源泉。

### ユーザー体験フロー（変更なし）

```
Safari で記事を読む
    ↓  (Share Sheet で保存)
アプリが自動で知識を抽出・タグ付け
    ↓
AI ブレイン画面を開く
    ↓
自分の知識カテゴリ分布が静かに見える
    ↓
また記事を読もうという気になる
```

### 5-2. Section 1 — Stats Row（統計行）

**目的**: 積み上げた総量を一目で確認する満足感

| 項目 | 表示内容 | データソース |
|---|---|---|
| 記事 | 保存した記事の総数 | `Article` 総件数 |
| 知識 | 抽出エンティティの重複排除数 | `KnowledgeEntity`（名前で重複除去） |
| ファクト | 抽出キーファクトの総数 | `KeyFact` 総件数 |

**UX ルール**:
- 3カラム等幅。数字が大きく（`.title2.bold`）、ラベルが小さい（`.caption`）
- 画面を開いたとき 0 から実数へカウントアップ（約 0.5 秒）
- Reduce Motion ON のときはカウントアップなし（即時表示）
- カードに薄い背景色（`dsCardBackground`）

**実装ファイル**: `AIBrainStatsRow.swift`（新規）

---

### 5-3. Section 2 — AI インサイトカード

**目的**: AI が自分の知識を客観的に報告してくれる感覚

| 状態 | 表示内容 |
|---|---|
| タグ 0 件 | 「まだ記事を保存していません。Safari から記事を保存しましょう」 |
| タグ 1 件以上 | 「最も読んでいる分野: {トップタグ名}（{記事数}記事）」 |

**UX ルール**:
- `sparkles` SF Symbol アイコン + テキスト
- AI ブランドカラー（薄い紫系）の背景と枠線
- タップ不可（情報表示のみ）

**実装ファイル**: `AIInsightCard.swift`（新規）

---

### 5-4. Section 3 — カテゴリ一覧

**目的**: 自分がどの分野に知識を積み上げているかの構造が見える

| 要素 | 詳細 |
|---|---|
| 並び順 | 記事数の多い順（降順）|
| 表示対象 | 記事が 1 件以上あるタグのみ |
| 各行の内容 | タグ名 ＋ プログレスバー ＋「N記事」 |
| プログレスバー | 最多タグを 100% として比率計算 |
| タップ動作 | そのタグで絞り込んだ記事一覧（`TagFilteredListView`）へ遷移 |
| タグ 0 件時 | `ContentUnavailableView`「カテゴリがありません」 |

**プログレスバーの計算**:
```
バー幅 = (このタグの記事数 / 最多タグの記事数) × 利用可能な最大幅
```

**実装ファイル**: `KnowledgeCategoryRow.swift`（新規）

## 6. ストレスゼロ原則（v1 継承 + v2 廃止項目追記）

### やること

| 要素 | 内容 |
|---|---|
| 累計カウント | 記事数・知識数・ファクト数を常時表示（増え続ける安心感）|
| カウントアップ | 画面起動時に 0 → 実数へアニメーション |
| カテゴリ可視化 | タグ別プログレスバーを眺めること自体が達成感 |

### やらないこと

| 要素 | 理由 |
|---|---|
| レベル数字（Lv.5 など） | ストレスになる |
| バッジ・実績通知 | ストレスになる |
| ストリーク（連続日数） | ストレスになる |
| 先週比・増減表示 | ストレスになる |
| ランキング・他者比較 | ストレスになる、スコープ外 |
| ネットワークグラフ | 意味が伝わらない（v2 で廃止）|
| 横スクロールカード群 | 構造が分散する（v2 で廃止）|
| 通知・ポップアップ演出 | ストレスになる |

## 7. データマッピング（v2 更新版）

既存のデータモデルで全要件をまかなえる。**新規スキーマ追加不要。**

| 画面要素 | 使うモデル / プロパティ |
|---|---|
| Stats Row — 記事数 | `Article` 総件数 |
| Stats Row — 知識数 | `KnowledgeEntity.name`（小文字+トリム後の重複除去）|
| Stats Row — ファクト数 | `KeyFact` 総件数 |
| AI インサイトカード | `Tag`（記事数最多の 1 件）|
| カテゴリ一覧（行） | `Tag`（`tag.articles.count > 0` の全件）|
| カテゴリ順序 | `tag.articles.count` の降順（インメモリソート）|
| プログレスバー比率 | `tag.articles.count / 最多タグ.articles.count` |
| タップ遷移 | `TagFilteredListView(tagName: tag.name)` |

**注**: SwiftData は Relationship のカウントによるソートをサポートしないため、`@Query` でタグ全件取得後にインメモリソートする。
