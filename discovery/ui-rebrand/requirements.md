# UI リブランディング & AI ブレインタブ — 要件定義書

**作成日**: 2026-05-05  
**ステータス**: 確定（実装依頼可能）  
**対象**: 新規タブ追加 + アプリリブランディング  
**既存コードへの影響**: 最小限（新ファイル追加が中心）

---

## 1. コアコンセプト

> **「Safari で読んだ記事が自動で吸収され、あなた専用の iPhone AI がどんどん育っていく」**

ユーザーは意識せず知識を蓄積し、ふと画面を開くと自分の AI が成長している。  
その成長を確認する快感がモチベーションの源泉。

### ユーザーの体験フロー

```
Safari で記事を読む
    ↓  (Share Sheet で保存)
アプリが自動で知識を抽出・タグ付け
    ↓
AI ブレイン画面を開く
    ↓
「また育ってる」と気づいて嬉しくなる
    ↓
また記事を読もうという気になる
```

## 3. タブ構成（推奨：2タブ）

### 現在

```
NavigationStack 1本（タブなし）
└── ArticleListView（記事一覧）
     ├── TagListView（ナビゲーション遷移）
     └── ArticleDetailView（シート）
```

### 変更後

```
TabView
├── Tab 1: ライブラリ  📚
│    └── ArticleListView（現状のまま、改修なし）
└── Tab 2: AI ブレイン  🧠
     └── [新規] AIBrainView（本書で定義）
```

**設計方針**:
- タグ一覧は独立タブにしない → AI ブレイン画面内のノードタップで遷移
- 既存の `ArticleListView` は一切改修しない
- `KnowledgeTreeApp.swift` の `WindowGroup` 内を `TabView` でラップするだけ

## 4. 新画面「AI ブレイン」— 詳細仕様

### 4-1. レイアウト全体図

```
┌──────────────────────────────────────┐
│  NavigationBar: "AI ブレイン"        │
├──────────────────────────────────────┤
│  [Section 1] AI パワーゲージ          │  高さ: 約160pt
│  ────────────────────────────────    │
│  グラデーションカード（ライト系）     │
│  ┌──────────────────────────────┐   │
│  │  ○○ 記事を吸収済              │   │
│  │  ○○ 知識  ·  ○○ キーファクト  │   │
│  │  "Your AI is growing"         │   │
│  └──────────────────────────────┘   │
│  （静かなパルスアニメーション）       │
├──────────────────────────────────────┤
│  [Section 2] ナレッジマップ           │  高さ: 残り全体
│  ────────────────────────────────    │
│  タグ = 円形ノード（大きさ = 記事数） │
│  エンティティ = ノード間の接続線      │
│  新ノード: フェードインで出現         │
│  タップ: そのタグの記事一覧へ遷移     │
│  ピンチ: ズームイン/アウト            │
├──────────────────────────────────────┤
│  [Section 3] 最近の動き               │  高さ: 約120pt
│  ────────────────────────────────    │
│  横スクロールカード群:                │
│  「今週 ○件 新たに吸収」             │
│  「最近育ったテーマ: ○○」            │
│  「新しい繋がり: ○○ ↔ ○○」          │
└──────────────────────────────────────┘
```

### 4-2. Section 1 — AI パワーゲージ

| 要素 | 仕様 |
|---|---|
| 背景 | iOS ネイティブ系グラデーション（ライト） |
| メイン数字 | `Article` 総数（「○○ 記事を吸収済」） |
| サブ数字 | `KnowledgeEntity` 数 + `KeyFact` 数 |
| サブテキスト | 固定文言「Your AI is growing」（英語でブランド感を出す） |
| アニメーション | 起動時に数字がカウントアップ（0 → 実数）、約0.6秒 |
| 更新タイミング | 新記事保存 → `RefreshTrigger` 経由で即時反映 |

### 4-3. Section 2 — ナレッジマップ

| 要素 | 仕様 |
|---|---|
| ノード | `Tag` 1件 = 円形ノード1個 |
| ノードサイズ | `tag.articles.count` に比例（最小 40pt / 最大 100pt） |
| ノードラベル | タグ名（ノード内または直下） |
| エッジ（接続線） | 同一 `KnowledgeEntity` を持つタグ同士を線で結ぶ |
| 新ノード演出 | 新しく作られたタグは 0.4秒 フェードイン |
| タップ動作 | `TagFilteredListView(tagName:)` へ遷移 |
| ジェスチャー | ピンチでズーム、ドラッグでパン |
| 実装方式 | `Canvas` + `GeometryReader`（SwiftUI ネイティブ）か `ForceLayout` ライブラリを検討 |
| タグゼロ時 | 「まだ記事がありません。Safari から記事を保存しよう！」のエンプティステート |

### 4-4. Section 3 — 最近の動き

| カード | データソース | 条件 |
|---|---|---|
| 今週 ○件 新たに吸収 | `Article.savedAt` | 直近7日間の件数 |
| 最近育ったテーマ | `Tag` + `Article.savedAt` | 直近7日で最も記事が増えたタグ Top3 |
| 新しい繋がり | `KnowledgeEntity` | 直近7日で初めて出現したエンティティ |

## 5. ゲーミフィケーション設計（ストレスゼロ原則）

### やること

| 要素 | 内容 |
|---|---|
| 累計カウント | 記事数・エンティティ数・キーファクト数を常時表示（増え続ける安心感） |
| 新ノード演出 | 新しいタグが静かにフェードイン |
| カウントアップ | 画面起動時に数字が0から実数へアニメーション |
| 繋がりの可視化 | マップを眺めること自体が達成感 |

### やらないこと

| 要素 | 理由 |
|---|---|
| レベル数字 (Lv.5 など) | ストレスになる |
| バッジ・実績通知 | ストレスになる |
| ストリーク（連続日数） | ストレスになる |
| ランキング・他者比較 | スコープ外（将来も未定） |
| 「○○ 解放！」系ポップアップ | ストレスになる |

## 6. データマッピング

既存のデータモデルで全要件をまかなえる。**新規スキーマ追加不要。**

| 画面要素 | 使うモデル / プロパティ |
|---|---|
| 吸収済み記事数 | `Article` 総件数 |
| 知識数 | `ExtractedKnowledge.entities.count`（全記事合計） |
| キーファクト数 | `ExtractedKnowledge.keyFacts.count`（全記事合計） |
| マップノード | `Tag`（全件） |
| ノードサイズ | `tag.articles.count` |
| エッジ（接続線） | `KnowledgeEntity.name` が一致するタグペア |
| 最近の動き | `Article.savedAt`、`Tag`、`KnowledgeEntity` |

## 7. 実装スコープ

### 今回実装する（本 spec の対象）

- `TabView` 導入（`KnowledgeTreeApp.swift` 改修）
- `AIBrainView.swift` 新規作成（Section 1 / 2 / 3）
- ナレッジマップ（Canvas ベース or ForceLayout）
- パワーゲージカード
- 最近の動きカード群
- アプリ名変更（Bundle Name / Display Name）

### 将来 spec として切り出す

| 機能 | 理由 |
|---|---|
| Safari から自動追跡（読んだら自動保存） | Safari Extension または Background App Refresh が別途必要。工数大 |
| ナレッジマップの高度な Force-Directed Layout | 初期は簡易グリッド or 円配置でも可 |
| タグ間のエッジ重みづけ（接続強度） | データモデル拡張が必要 |
