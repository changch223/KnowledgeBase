# KnowledgeTree コードレビューレポート評価ルーブリック 最終版

このルーブリックは、コードレビューまたはコードベース監査レポートを「改善計画に採用できる成果物」として評価するための基準です。

OpenAI 版をベースに、Claude 版の強みである観察ポイント、アンチパターン、客観性、不確実性、誤検出訂正、コンテキスト適合を統合しています。KnowledgeTree のように calm UX、spec 系譜、cross-extension 整合性、AI 層を持つプロダクトで、一般論レビューや AI ハルシネーションを弾くことを重視します。

満点は 100 点です。80 点以上、かつ採用最低基準と致命軸ガードを満たす場合のみ、改善計画の入力として採用できます。

---

## 1. 判定の原則

### 採用判定

| 点数 | 評価 | 判定 |
|---:|---|---|
| 90-100 | Excellent | 意思決定と実装計画にそのまま使える |
| 80-89 | Strong | 採用可能。軽微な補足で実装に進める |
| 70-79 | Useful but incomplete | 有用だが、計画前に補足調査が必要 |
| 60-69 | Weak | 参考情報にはなるが、重要な証拠や優先順位が不足 |
| 0-59 | Unreliable | 改善計画には使えない。再レビューが必要 |

### 採用最低基準

80 点以上でも、以下をすべて満たさない場合は採用不可または条件付き採用とします。

- レビュー範囲、調査方法、除外範囲が明確。
- P0/P1 が優先順位つきで列挙されている。
- P0/P1 に正確な証拠がある。
- P0/P1 の `証拠・影響・提案・検証` の揃い率が 80% 以上。
- 修正対象と次アクションが具体的。
- テスト・検証状況が書かれている。
- 未検証範囲と制約が書かれている。
- UI/UX 影響がコード品質とは別に扱われている。
- プロダクト哲学、spec、既存設計判断と矛盾する提案に根拠がある。

### 致命軸ガード

以下に該当する場合、合計点に関わらず判定を制限します。

| 条件 | 強制判定 |
|---|---|
| 軸 2 `証拠の具体性` が 8/15 未満 | Useful but incomplete 以下 |
| 軸 3 `技術的正確性` が 8/15 未満 | Useful but incomplete 以下 |
| 軸 4 `コンテキスト適合` が 5/10 未満 | Useful but incomplete 以下 |
| 軸 2 または軸 3 が 0 点 | Unreliable |
| 存在しない証拠、捏造テスト結果、捏造挙動が主要指摘に含まれる | Unreliable または失格 |
| P0/P1 の `証拠・影響・提案・検証` の揃い率が 80% 未満 | 条件付き採用以下 |
| 確認なしの CRITICAL/P0 断言が仕様や既存設計だった | Useful but incomplete 以下 |

## 2. 失格条件

以下のいずれかに該当するレポートは、点数に関係なく原則不合格です。

| 失格条件 | 確認手順 |
|---|---|
| 存在しないファイル、行番号、挙動、コマンド結果、テスト結果を記載している | `ls`、`rg`、対象ファイルの該当行、テストログで照合する |
| レビュー範囲を明記していない | 冒頭または付録に対象ディレクトリ、対象ファイル数、除外範囲があるか確認する |
| 主要な指摘に証拠がない | P0/P1 の各指摘に `file:line`、関数名、再現条件、コマンド結果のいずれかがあるか確認する |
| 「全部見た」と主張しているが、対象ファイル数・読み方・除外範囲を示していない | `git ls-files` や `rg --files` の総数と、レポート内の調査数を照合する |
| 抽象的な助言だけで、修正対象や次アクションがない | 各主要指摘に対象ファイル、期待挙動、テスト方針があるか確認する |
| 破壊的または高リスクな変更を、影響や代替案なしに推奨している | データ移行、削除、認証、保存形式変更、API 変更に代替案とロールバック案があるか確認する |
| レビュー対象コード上で明らかな重大ユーザー影響を見落としている | 主要ユーザーフロー、保存・同期、課金、認証、権限、拡張ターゲットを照合する |
| AI ハルシネーション疑いが濃い | file:line がずれる、存在しない API を前提にする、一般論だけでコード挙動を説明しない、などを複数確認する |

## 3. 採点サマリー

| 評価軸 | 点数 |
|---|---:|
| 1. スコープと調査方法の透明性 | 10 |
| 2. 証拠の具体性と追跡可能性 | 15 |
| 3. 技術的正確性 | 15 |
| 4. コンテキスト適合 | 10 |
| 5. 深刻度と優先順位 | 10 |
| 6. 改善提案の実行可能性 | 10 |
| 7. テストと検証観点 | 10 |
| 8. UI/UX とプロダクト影響 | 10 |
| 9. 客観性と訂正規律 | 5 |
| 10. 伝達品質 | 5 |
| 合計 | 100 |

### 重みの考え方

- 証拠と技術的正確性は合計 30 点。AI 生成レビューで最も危険な失敗が、もっともらしい誤指摘や捏造証拠だからです。
- コンテキスト適合は 10 点。KnowledgeTree では calm UX、spec、既存設計判断に沿わない一般論レビューを採用しないためです。
- UI/UX とテストは独立軸。コード品質だけでなく、ユーザー成功と回帰防止に直結するためです。
- 客観性は 5 点ですが、低い場合は判定を下げます。レビューの信頼性を支える基礎だからです。

プロジェクト特性に応じて配点は調整できます。例: セキュリティ重視なら軸 3 を 20 点、軸 10 を 0-3 点へ圧縮する。UX 重視なら軸 8 を 15 点、軸 6 を 5 点へ圧縮する。

## 4. 証拠ラベル

主要指摘、特に P0/P1 には証拠ラベルを付けます。

| ラベル | 意味 |
|---|---|
| `直接確認` | 対象ファイルと該当行を読んで確認した |
| `grep確認` | `rg` などでコードベース横断の存在や不在を確認した |
| `テスト実行済み` | 実際にテスト、ビルド、lint、静的解析を実行した |
| `手動再現` | ユーザーフローや UI 操作で再現した |
| `推論` | コードから合理的に推論したが、実行検証はしていない |
| `未検証` | 可能性として提示しているが、確認は未完了 |
| `外部報告` | サブエージェント、人間レビュー、既存レポートから受け取った情報 |

`推論`、`未検証`、`外部報告` は、そのまま P0/P1 の確定指摘にしてはいけません。P0/P1 にする場合は、追加確認または不確実性の明示が必要です。

## 5. 評価軸

### 軸 1. スコープと調査方法の透明性 - 10点

レビュー対象と調査方法が明確かを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | スコープがない |
| 3 | 「アプリ全体を確認」など表現が曖昧 |
| 6 | 主要ディレクトリは列挙しているが、読んだファイル・機械的に走査したファイル・未検証領域の区別がない |
| 8 | 対象ディレクトリ、ファイル種別、主なコマンド、除外範囲を明記している |
| 10 | 対象範囲、調査方法、限界、各領域の信頼度まで明記している |

観察ポイント:

- 対象ディレクトリ、対象ファイル数、対象行数があるか。
- 直接読んだファイルと、機械的に検索しただけのファイルを区別しているか。
- 未実行のビルド、テスト、手動検証を隠していないか。
- 生成物、依存ライブラリ、アセット、外部サービスなどの除外範囲が明確か。
- 「全部見た」という主張に検証可能な根拠があるか。

### 軸 2. 証拠の具体性と追跡可能性 - 15点

各指摘が、再確認可能な根拠に結びついているかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | 根拠がない、または主要証拠が捏造されている |
| 5 | ファイル名はあるが、行番号・関数名・再現条件が不足している |
| 9 | 多くの指摘に参照はあるが、一部が広すぎる |
| 12 | ファイル、行番号、関数、コードパターン、コマンド結果、テスト名などが具体的 |
| 15 | 重要指摘のすべてに、正確な証拠・再現手順・推論の流れがある |

観察ポイント:

- `path/to/File.swift:123` のような具体参照があるか。
- file:line が、実際の該当コードと一致しているか。
- コード引用、関数名、状態遷移、データフロー、ユーザーフローが指摘とつながっているか。
- 実行コマンドと結果要約があるか。
- 事実、推論、仮説、未検証事項が分離されているか。
- 同じ主張を `直接確認` と `grep確認` など複数手段で補強しているか。

### 軸 3. 技術的正確性 - 15点

指摘が実際のコード挙動を正しく説明しているかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | 主要指摘の多くが誤り、または存在しない挙動を前提にしている |
| 4 | もっともらしいが検証不足の推測が多い |
| 8 | 大半は正しいが、一部に過剰な断定や弱い仮説がある |
| 12 | 事実と推論を分け、フレームワークや既存設計を踏まえている |
| 15 | 正確で、例外条件・既存アーキテクチャ・プラットフォーム制約まで考慮している |

観察ポイント:

- 実コードの挙動を説明しているか。
- バグ、技術的負債、仕様上のトレードオフ、UX 不整合を区別しているか。
- SwiftUI の `@State`、`@Query`、`@Observable`、`@Environment` などの役割を取り違えていないか。
- SwiftData の `@Model`、Relationship、削除規則、migration、CloudKit 制約を現コード通りに扱っているか。
- Extension、Widget、Share Extension、Safari Extension の Bundle 隔離や権限境界を理解しているか。
- AI 層がある場合、プロンプト、トークン上限、ループ制御、ハルシネーション対策、Foundation Models の制約を踏まえているか。
- コード上は正しくても、運用、スケール、アクセシビリティで問題化する点を適切に扱っているか。

### 軸 4. コンテキスト適合 - 10点

プロダクト哲学、開発段階、spec 系譜、既存設計判断を尊重しているかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | 一般論を押し付け、プロジェクト方針や明示制約を無視している |
| 3 | コードは見ているが、設計判断や spec 履歴への関心が薄い |
| 6 | CLAUDE.md、DESIGN.md、Constitution、spec、既存 docs の一部を参照している |
| 8 | 既存設計判断を尊重し、その文脈の中で改善案を出している |
| 10 | プロダクト哲学とトレードオフを明示し、設計意図を壊さない代替案まで示している |

観察ポイント:

- calm UX を採用している箇所に、安易な alert や過剰通知を推奨していないか。
- spec で意思決定済みの挙動を、根拠なくバグ扱いしていないか。
- 既存のナビゲーション、デザインシステム、保存モデル、拡張ターゲットの責務境界に沿っているか。
- 「業界標準」と「このプロダクトの方針」を区別しているか。
- 提案がチームの現在フェーズ、リリースリスク、移行余力に合っているか。

### 軸 5. 深刻度と優先順位 - 10点

指摘の重要度が、ユーザー影響、発生可能性、影響範囲、修正順序に基づいて整理されているかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | 優先順位がない |
| 3 | ラベルはあるが基準が一貫していない |
| 6 | 大まかな優先順位はあるが、軽微な問題と重大問題が混ざる |
| 8 | P0/P1/P2/P3 が、影響と発生可能性に基づいている |
| 10 | 影響、発生可能性、影響範囲、修正コスト、推奨修正順まで示している |

推奨する深刻度基準:

- P0: 主要ユーザーフロー破綻、データ消失、クラッシュ、セキュリティ/プライバシーリスク、削除済み挙動をテストが検証している状態。
- P1: 信頼性、保守性、パフォーマンス、UX 信頼感に大きく影響する問題。
- P2: ローカライズ、観測性、アクセシビリティ、設計整理、将来の拡張性に関わる改善。
- P3: あると良い改善、軽微な整理、表現の磨き込み。

半定量評価:

| 項目 | 低 | 中 | 高 |
|---|---|---|---|
| 影響 | 内部品質のみ | 一部ユーザーに影響 | 主要フロー、データ、信頼に影響 |
| 発生可能性 | まれ | 条件付きで発生 | 通常利用で発生 |
| 影響範囲 | 単一画面 | 複数画面または一部ターゲット | アプリ全体または複数ターゲット |
| 修正コスト | 小 | 中 | 大 |

P0/P1 には、少なくとも影響と発生可能性の説明が必要です。

### 軸 6. 改善提案の実行可能性 - 10点

チームが迷わず実装計画に落とし込めるかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | 抽象論だけ |
| 3 | 方向性はあるが、対象ファイルや実装方針が不足している |
| 6 | 多くの指摘に修正方針がある |
| 8 | 対象ファイル、期待挙動、テスト方針が明確 |
| 10 | 実装順序、影響範囲、移行方法、回帰リスク、担当境界まで具体的 |

観察ポイント:

- 変更対象ファイルまたはモジュールを示しているか。
- 修正後のユーザー体験またはシステム挙動を示しているか。
- 追加・更新すべきテストを示しているか。
- 既存アーキテクチャと整合しているか。
- 大きな変更では段階的移行案を出しているか。
- 代替案とトレードオフを示しているか。
- Quick Win と設計改善を分けているか。

### 軸 7. テストと検証観点 - 10点

既存テストの状態、検証済み範囲、未検証リスクを扱えているかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | テストや検証に触れていない |
| 3 | 「テストを追加するべき」程度 |
| 6 | 不足テストや古いテストを一部特定している |
| 8 | 主要指摘ごとに追加・更新すべきテストを対応づけている |
| 10 | 実行コマンド、未実行理由、環境制約、古いテスト、手動検証計画まで示している |

確認すべき検証領域:

- Unit Tests
- UI Tests
- SwiftData/CloudKit 保存・同期
- Share Extension
- Safari Extension
- Widget
- ローカライズ検証
- アクセシビリティ検証
- ビルド、Lint、静的解析
- 主要ユーザーフローの手動検証

観察ポイント:

- 実行したテストと、実行していないテストが分かるか。
- 未実行理由が環境制約なのか、時間制約なのか、対象外なのか明確か。
- 古いテスト、削除済み UI を参照するテスト、壊れた accessibility identifier を見つけているか。
- P0/P1 ごとに回帰防止テストが提案されているか。

### 軸 8. UI/UX とプロダクト影響 - 10点

コード上の問題を、ユーザーの成功、迷い、信頼、継続利用に結びつけて分析できているかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | UI/UX 観点がない |
| 3 | 見た目の問題だけを軽く触れている |
| 6 | 主要なユーザー混乱や導線不備を特定している |
| 8 | 初回体験、ナビゲーション、状態表示、エラー、アクセシビリティ、信頼感に結びつけている |
| 10 | タスク成功率、認知負荷、情報設計、空状態、復帰導線、サポート負荷まで含めて提案している |

観察ポイント:

- オンボーディング、空状態、エラー状態で次の行動が分かるか。
- リンクやボタンが反応するように見えるのに遷移しないなど、期待違反を拾っているか。
- 長時間処理に進捗、成功、失敗フィードバックがあるか。
- VoiceOver、Dynamic Type、Reduce Motion、キーボード操作で破綻しないか。
- calm UX を保ちながら、必要な場面でだけ状態を surface する提案になっているか。
- UI 不整合をコード品質と混ぜず、ユーザー影響として説明しているか。

### 軸 9. 客観性と訂正規律 - 5点

レビューが否定一辺倒ではなく、良い点、不確実性、誤検出訂正を扱えているかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | 否定的のみ、良い点ゼロ、確認なしの断言が多い |
| 1 | 表面的な賞賛はあるが、不確実性や限界を明示していない |
| 3 | 良い点を具体的に挙げ、不確実性を明示している |
| 5 | 誤検出訂正、レビュー者の限界、対立解釈、追加確認が必要な箇所を独立して扱っている |

観察ポイント:

- 良い実装を最低 3 つ、具体的な箇所やパターンに紐づけて挙げているか。
- 確信度の違う主張が同じ語気で書かれていないか。
- 「仕様かもしれない」「未検証」「追加確認が必要」を隠していないか。
- 誤検出が見つかった場合、訂正セクションで率直に扱っているか。
- レビュー対象への敬意があり、人格批判や曖昧な批判に流れていないか。

### 軸 10. 伝達品質 - 5点

レポートが読みやすく、意思決定に使いやすいかを評価します。

| 点数 | 基準 |
|---:|---|
| 0 | 構造がなく読みにくい |
| 1 | 意味は分かるが、冗長または分類が弱い |
| 3 | 要点、証拠、優先順位、提案が整理されている |
| 5 | 最重要リスクから始まり、事実・推論・提案が分かれ、ロードマップとして使える |

良いレポート構成:

- 先頭にエグゼクティブサマリーを置く。
- P0/P1 を先に出す。
- 各指摘に `証拠`、`影響`、`提案`、`検証` を含める。
- 事実と推測を分ける。
- 曖昧な批判ではなく、改善可能な表現にする。
- 最後に推奨実装順と未検証事項を置く。

## 6. P0/P1 揃い率

P0/P1 指摘は、以下 4 要素を必ず確認します。

- 証拠: `file:line`、関数名、再現手順、コマンド結果など。
- 影響: ユーザー、データ、セキュリティ、信頼性、開発速度への影響。
- 提案: 対象ファイル、修正方針、代替案。
- 検証: 追加・更新すべきテスト、手動確認、未検証リスク。

計算式:

```text
P0/P1 揃い率 = 揃っている要素数 / (P0/P1 指摘数 × 4)
```

採用条件:

- 90% 以上: 良好。
- 80-89%: 採用可。軽微な補足でよい。
- 60-79%: 条件付き採用。実装前に補足が必要。
- 59% 以下: 採用不可。P0/P1 の再検証が必要。

## 7. AI 生成レビュー特有のアンチパターン

以下が複数見られる場合、AI ハルシネーション疑いとして失格条件を検討します。

| パターン | 症状 | 対処 |
|---|---|---|
| 文章は整っているが file:line がない | 読みやすいが再確認できない | 軸 2 を下げる。P0/P1 は採用しない |
| 過剰に網羅的だが各指摘が浅い | テンプレ生成の可能性 | 主要指摘だけ再検証し、浅い項目を P2/P3 以下へ落とす |
| 行番号が数行ずれる | LLM の位置参照幻覚 | 実ファイルの該当行で照合する |
| 「Swift では一般的に」など普遍命題が多い | 現コードの説明になっていない | 軸 3 と軸 4 を下げる |
| エラーハンドリングや null チェックの一般指摘が多い | 分量稼ぎの可能性 | 実コードの具体挙動に紐づくものだけ残す |
| 存在しない API や制約を前提にする | 技術的誤り | 軸 3 を大きく下げる。主要指摘なら失格 |
| プロダクト哲学を知らずに UX を断定する | calm UX や spec と衝突 | 軸 4 と軸 8 を下げる |

## 8. アンチパターン早見表

| アンチパターン | 該当軸 | 対処 |
|---|---|---|
| 全 P0 / 全 HIGH | 軸 5 | 影響と発生可能性を二軸で分離する |
| 重大度ラベルがない | 軸 5 | P0-P3 または Critical-Low を導入する |
| file:line がゼロ | 軸 2 | 証拠を付け直す。主要指摘は採用しない |
| 「業界標準」絶対視 | 軸 4 | プロダクト哲学、spec、既存設計判断を先に確認する |
| 良い点ゼロ | 軸 9 | 優秀な実装を最低 3 つ具体的に列挙する |
| 「リファクタすべき」で止まる | 軸 6 | 対象ファイル、修正後挙動、テストの 3 点を必須にする |
| UI だけ深掘りしてセキュリティやデータ層が未調査 | 軸 1、7、8 | 調査範囲と未調査領域を明示する |
| テスト追加だけ提案し、既存テストを見ていない | 軸 7 | 既存テスト、古いテスト、未実行コマンドを確認する |
| 仕様かバグかを区別しない | 軸 3、4、9 | spec と既存 docs を照合し、不確実性を明示する |
| 破壊的変更を即推奨 | 軸 4、6 | 移行案、代替案、ロールバック、回帰リスクを要求する |

## 9. 比較レビューモード

同じコードベースに対する複数レビュー、例えば Claude、GPT、人間レビューを比較する場合に使います。

### 比較手順

1. 各レビューをこのルーブリックで個別採点する。
2. P0/P1 指摘を一覧化し、重複、矛盾、片方だけの指摘を分ける。
3. 矛盾する指摘は、証拠ラベルと file:line で再検証する。
4. 両方が指摘している P0/P1 は優先度を上げる。
5. 片方だけの指摘は、証拠の強さとコンテキスト適合で採否を決める。

### 比較採点シート

| 指摘 | レビュー A | レビュー B | 証拠 | 判定 | 次アクション |
|---|---|---|---|---|---|
|  | P0/P1/P2/P3 | P0/P1/P2/P3 | file:line / コマンド | 採用 / 保留 / 却下 |  |

## 11. 運用メモ

- 作成時は、レビュー本文のテンプレートに `証拠`、`影響`、`提案`、`検証` を最初から入れる。
- 採点時は、まず失格条件と致命軸ガードを見る。点数計算はその後でよい。
- P0/P1 は少なくとも 1 件ずつ file:line を実ファイルで照合する。
- 評価者が迷ったら、点数を上げるより `条件付き採用` にして追加検証を明示する。
- 自己採点は楽観的に振れやすいので、重要なレビューは第三者または別モデルで再採点する。
- ルーブリック自体は固定しすぎない。重大な誤採用または誤却下が起きたら、失格条件と観察ポイントを更新する。
