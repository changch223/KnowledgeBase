# コードレビューレポート評価ルーブリック

このルーブリックは、コードレビューまたはコードベース監査レポートの品質を評価するための基準です。目的は、レポートが「実際の改善計画に使える成果物」になっているかを、証拠・正確性・優先順位・実行可能性・UI/UX影響まで含めて判定することです。

満点は100点です。80点以上を、改善計画の入力として採用できる最低ラインとします。

## 失格条件

以下のいずれかに該当するレポートは、点数に関係なく不合格とします。

- 存在しないファイル、行番号、挙動、コマンド結果、テスト結果を記載している。
- レビュー範囲を明記していない。
- 主要な指摘に証拠がない。
- 「全部見た」と主張しているが、対象ファイル数・読み方・除外範囲を示していない。
- 抽象的な助言だけで、修正対象や次アクションがない。
- 破壊的または高リスクな変更を、影響や代替案なしに推奨している。
- レビュー対象コード上で明らかな重大ユーザー影響を見落としている。

## 採点サマリー

| 評価軸 | 点数 |
|---|---:|
| 1. スコープと調査方法の透明性 | 10 |
| 2. 証拠の具体性 | 15 |
| 3. 技術的正確性 | 20 |
| 4. 深刻度と優先順位 | 10 |
| 5. 改善提案の実行可能性 | 15 |
| 6. テストと検証観点 | 10 |
| 7. UI/UXとプロダクト影響 | 10 |
| 8. 伝達品質 | 10 |
| 合計 | 100 |

## 1. スコープと調査方法の透明性 - 10点

レビュー対象と調査方法が明確かを評価します。

- 0点: スコープがない。
- 3点: 「アプリ全体を確認」など表現が曖昧。
- 6点: 主要ディレクトリは列挙しているが、読んだファイル・機械的に走査したファイル・未検証領域の区別がない。
- 8点: 対象ディレクトリ、ファイル種別、主なコマンド、除外範囲を明記している。
- 10点: 対象範囲、調査方法、限界、各領域の信頼度まで明記している。

高評価の証拠例:

- 対象ディレクトリ一覧
- 対象ファイル数と行数
- 実行した検索・解析コマンド
- 読み込んだ主要ファイル
- 未実行のビルド、テスト、手動検証
- 除外した生成物、依存ライブラリ、アセット

## 2. 証拠の具体性 - 15点

各指摘が、再確認可能な根拠に結びついているかを評価します。

- 0点: 根拠がない。
- 5点: ファイル名はあるが、行番号・関数名・再現条件が不足している。
- 9点: 多くの指摘に参照はあるが、一部が広すぎる。
- 12点: ファイル、関数、コードパターン、コマンド結果、テスト名などが具体的。
- 15点: 重要指摘のすべてに、正確な証拠・再現手順・推論の流れがある。

高評価の証拠例:

- `path/to/File.swift:123` のような具体参照
- 実行コマンドと結果要約
- ユーザーフロー上の再現手順
- データフロー、状態遷移、非同期処理の説明
- 現行UI、テスト、仕様文書の不一致

## 3. 技術的正確性 - 20点

指摘が実際のコード挙動を正しく説明しているかを評価します。

- 0点: 主要指摘の多くが誤り。
- 5点: もっともらしいが検証不足の推測が多い。
- 10点: 大半は正しいが、一部に過剰な断定や弱い仮説がある。
- 15点: 事実と推論を分け、フレームワークや既存設計を踏まえている。
- 20点: 正確で、例外条件・既存アーキテクチャ・プラットフォーム制約まで考慮している。

評価観点:

- 実コードの挙動を説明しているか。
- バグ、技術的負債、仕様上のトレードオフ、UX不整合を区別しているか。
- SwiftUI、SwiftData、CloudKit、Extension、Widgetなど、対象技術の制約を踏まえているか。
- コード上は正しくても、運用・スケール・アクセシビリティで問題化する点を適切に扱っているか。

## 4. 深刻度と優先順位 - 10点

指摘の重要度が、ユーザー影響・発生可能性・修正順序に基づいて整理されているかを評価します。

- 0点: 優先順位がない。
- 3点: ラベルはあるが基準が一貫していない。
- 6点: 大まかな優先順位はあるが、軽微な問題と重大問題が混ざる。
- 8点: P0/P1/P2などが、影響と発生可能性に基づいている。
- 10点: 影響、発生可能性、影響範囲、推奨修正順まで示している。

推奨する深刻度基準:

- P0: 主要ユーザーフロー破綻、データ消失、クラッシュ、セキュリティ/プライバシーリスク、テストが削除済み挙動を検証している状態。
- P1: 信頼性、保守性、パフォーマンス、UX信頼感に大きく影響する問題。
- P2: ローカライズ、観測性、アクセシビリティ、設計整理、将来の拡張性に関わる改善。
- P3: あると良い改善、軽微な整理、表現の磨き込み。

## 5. 改善提案の実行可能性 - 15点

チームが迷わず実装計画に落とし込めるかを評価します。

- 0点: 抽象論だけ。
- 5点: 方向性はあるが、対象ファイルや実装方針が不足している。
- 9点: 多くの指摘に修正方針がある。
- 12点: 対象ファイル、期待挙動、テスト方針が明確。
- 15点: 実装順序、影響範囲、移行方法、回帰リスク、担当境界まで具体的。

良い改善提案の条件:

- 変更対象ファイルまたはモジュールを示す。
- 修正後のユーザー体験またはシステム挙動を示す。
- 追加・更新すべきテストを示す。
- 既存アーキテクチャと整合している。
- 大きな変更では段階的移行案を出す。
- 代替案とトレードオフを示す。

## 6. テストと検証観点 - 10点

既存テストの状態、検証済み範囲、未検証リスクを扱えているかを評価します。

- 0点: テストや検証に触れていない。
- 3点: 「テストを追加するべき」程度。
- 6点: 不足テストや古いテストを一部特定している。
- 8点: 主要指摘ごとに追加・更新すべきテストを対応づけている。
- 10点: 実行コマンド、未実行理由、環境制約、古いテスト、手動検証計画まで示している。

確認すべき検証領域:

- Unit Tests
- UI Tests
- SwiftData/CloudKit保存・同期
- Share Extension
- Safari Extension
- Widget
- ローカライズ検証
- アクセシビリティ検証
- ビルド、Lint、静的解析
- 主要ユーザーフローの手動検証

## 7. UI/UXとプロダクト影響 - 10点

コード上の問題を、ユーザーの成功・迷い・信頼・継続利用に結びつけて分析できているかを評価します。

- 0点: UI/UX観点がない。
- 3点: 見た目の問題だけを軽く触れている。
- 6点: 主要なユーザー混乱や導線不備を特定している。
- 8点: 初回体験、ナビゲーション、状態表示、エラー、アクセシビリティ、信頼感に結びつけている。
- 10点: タスク成功率、認知負荷、情報設計、空状態、復帰導線、サポート負荷まで含めて提案している。

高評価のUI/UX指摘例:

- オンボーディング文言が現行タブと不一致。
- リンクやボタンが反応するように見えるが遷移しない。
- 設定が重複し、どちらが正しいか分からない。
- 長時間処理に進捗・成功・失敗フィードバックがない。
- 空状態やエラー状態で次に何をすればよいか分からない。
- VoiceOverやDynamic Typeで破綻する可能性がある。
- テストのアクセシビリティ識別子が現行UIとずれている。

## 8. 伝達品質 - 10点

レポートが読みやすく、意思決定に使いやすいかを評価します。

- 0点: 構造がなく読みにくい。
- 3点: 意味は分かるが、冗長または分類が弱い。
- 6点: セクション分けされているが、重要情報が埋もれている。
- 8点: 要点、証拠、優先順位、提案が整理されている。
- 10点: 最重要リスクから始まり、事実・推論・提案が分かれ、ロードマップとして使える。

良いレポート構成:

- 先頭にエグゼクティブサマリーを置く。
- P0/P1を先に出す。
- 各指摘に「証拠」「影響」「提案」「検証」を含める。
- 事実と推測を分ける。
- 曖昧な批判ではなく、改善可能な表現にする。
- 最後に推奨実装順と未検証事項を置く。

## 総合評価

| 点数 | 評価 | 判定 |
|---:|---|---|
| 90-100 | Excellent | 意思決定と実装計画にそのまま使える。 |
| 80-89 | Strong | 採用可能。軽微な補足で実装に進める。 |
| 70-79 | Useful but incomplete | 有用だが、計画前に補足調査が必要。 |
| 60-69 | Weak | 参考情報にはなるが、重要な証拠や優先順位が不足。 |
| 0-59 | Unreliable | 改善計画には使えない。再レビューが必要。 |

## 採用最低基準

採用可能なレポートは80点以上、かつ以下をすべて満たす必要があります。

- レビュー範囲が明確。
- P0/P1が優先順位つきで列挙されている。
- P0/P1に証拠がある。
- 修正対象と次アクションが具体的。
- テスト・検証状況が書かれている。
- 未検証範囲と制約が書かれている。
- UI/UX影響がコード品質とは別に扱われている。

## ボーナス評価

以下を満たす場合、合計点が100点を超えない範囲で最大5点まで加点できます。

- 実装順序を示している。
- Quick Winと設計改善を分けている。
- 回帰リスクを示している。
- テストマトリクスを含む。
- プロダクト目標やユーザーフローに紐づけている。
- 複数のレビュー結果を比較し、矛盾点を検証している。

## 評価チェックリスト

レポートを評価するときは、以下を確認します。

- 何をどこまで見たかが明確か。
- 「全部見た」という主張に、ファイル数・方法・限界が伴っているか。
- 重要指摘は再現可能または証拠つきか。
- P0は本当にP0か。
- 軽微な指摘が重大指摘を埋もれさせていないか。
- 推奨修正は既存設計に沿っているか。
- 追加・更新すべきテストが具体的か。
- UI/UX影響をコード品質とは別軸で扱っているか。
- 現在の欠陥と将来改善を区別しているか。
- 実行した検証と実行していない検証が分かるか。
- 読者が次に何をすればよいか分かるか。
