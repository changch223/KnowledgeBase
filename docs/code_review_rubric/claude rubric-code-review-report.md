# コードレビュー報告書 評価ルーブリック

- **目的**: コードレビュー報告書（人間 / AI / 混成いずれの作成者でも）の品質を、観測可能な基準で評価する。
- **適用範囲**: 静的解析ベースのレビュー報告書（プルリクレビュー単発ではなく、機能横断・コードベース全体に対する形式の報告書）。
- **使い方**: 6 軸 × 4 段階で採点（最大 24 点）→ 総合判定。レッドフラグに該当する場合は段階を下げる。

---

## 1. 評価次元（6 軸）

| 軸 | 何を測るか | 失敗時の症状 |
|---|---|---|
| **1. エビデンス品質** | 主張の裏付け（file:line / コード引用 / 検証経路） | 「思う / 感じる」の積み重ねで再現できない |
| **2. 重大度キャリブレーション** | 影響 × 発生確率での区分、データ損失と style nit を混同しない | 全部「CRITICAL」、優先順位が無い |
| **3. 網羅性** | 主要領域を漏らさず確認、未調査領域を明示 | UI だけ / セキュリティだけ等の偏り、盲点に無自覚 |
| **4. コンテキスト適合** | プロダクト哲学・開発段階・明示制約の尊重 | 一般論の押し付け、設計意図を無視 |
| **5. 実行可能性 & 構造** | 具体的な修正方針 + 優先順位 + スキャナブル | 「より良く」抽象論、フラットな箇条書きで全体俯瞰不能 |
| **6. 客観性** | 良い点の認識、断定の回避、誤検出の率直な訂正 | 否定的のみ、確認なしの断言、誤検出が放置 |

---

## 2. 各軸の段階基準

各軸 4 段階。**3 = 達成水準（出荷可）**、4 = 卓越。

### 軸 1: エビデンス品質

| 段階 | 観察される特徴 |
|---|---|
| **1 不十分** | ファイル名すらない一般論。「エラーハンドリングが弱い」のような断言のみ |
| **2 改善余地** | ファイル名はあるが行番号なし。コード引用なし。出典が「読んだ感じ」 |
| **3 達成** | `file_path:line_number` 付き、関連コード抜粋あり、確認手段（Read / grep）を明示 |
| **4 卓越** | 検証経路を明示（自分で読んだ / grep / サブエージェント報告）。**不確実性を率直に表明**し、後から検証する読者の手がかりを残す |

**観察ポイント**:
- file:line が**何件のうち何件**に付いているか
- 「直接読んだ」「grep で確認」「報告を受けた」が区別されているか
- 同じ主張を裏付ける独立した手段（読 + grep）があるか

---

### 軸 2: 重大度キャリブレーション

| 段階 | 観察される特徴 |
|---|---|
| **1 不十分** | 全項目フラット、severity 表記なし。または全部「重要」 |
| **2 改善余地** | severity あるが過剰インフレ（「全 CRITICAL」「全 HIGH」）。style nit と data loss が同列 |
| **3 達成** | 明確な区分（P0〜P3 / Critical〜Low）、影響の大きさで分類、なぜその段階かの根拠あり |
| **4 卓越** | 影響 × 発生確率 × 修正コストの**多次元評価**、優先順位表で全体俯瞰、修正工数も見積もり |

**観察ポイント**:
- 「ユーザーデータが消える」と「変数名がイマイチ」が同じレベルで並んでいないか
- 「CRITICAL」「HIGH」を使うときに**なぜ**その段階かの説明があるか
- 出荷リスク / セキュリティ / UX / 技術的負債が分離されているか

---

### 軸 3: 網羅性

**標準チェックリスト**（プラットフォームで調整）:

| 領域 | 例 |
|---|---|
| セキュリティ・プライバシー | SSRF / 入力検証 / 権限スコープ / トラッキング |
| データ層 | スキーマ / マイグレーション / 整合性 / 削除規則 |
| 並行性 | Actor 隔離 / メインスレッドブロッキング / 競合 |
| エラーハンドリング | サイレント失敗 / 強制 unwrap / 空 catch |
| UI / View | パフォーマンス / 状態管理 / 構造 |
| アクセシビリティ | Dynamic Type / VoiceOver / Reduce Motion |
| テスト | カバレッジ / flaky / 不足領域 |
| ローカライズ | キー統一 / ハードコード |
| 拡張 / 周辺ターゲット | Widget / Share Ext / Safari Ext / 跨プロセス整合 |
| パフォーマンス | 起動時間 / レンダリング / メモリ |
| AI / 機械学習層（該当時） | プロンプト / トークン上限 / ハルシネーション / ループ制御 |

| 段階 | 観察される特徴 |
|---|---|
| **1 不十分** | 1-2 領域のみ |
| **2 改善余地** | 半数程度、各領域は表面的 |
| **3 達成** | 主要領域を網羅、各領域で具体例 |
| **4 卓越** | アプリ固有の領域（cross-process / AI / CloudKit 等）まで踏み込み、**未調査領域を明示**して読者に残課題を渡す |

---

### 軸 4: コンテキスト適合

| 段階 | 観察される特徴 |
|---|---|
| **1 不十分** | 一般論の押し付け、プロジェクト方針を無視（例: "calm UX" 採用に「全部 alert 出せ」） |
| **2 改善余地** | コードは見たが、なぜそうなっているか（設計判断・spec 履歴）に無関心 |
| **3 達成** | CLAUDE.md / DESIGN.md / spec 番号を参照、設計意図を尊重 |
| **4 卓越** | プロダクト哲学の**中で**改善を提案、トレードオフを明示（「calm UX を保ちつつ、能動操作だけ surface」など） |

**観察ポイント**:
- 「業界一般のベストプラクティス」と「このプロジェクトの方針」を区別しているか
- 既存の設計判断に対して「なぜそうなっているか」を理解してから否定しているか
- 提案が**そのチーム / プロダクトで実行可能**か

---

### 軸 5: 実行可能性 & 構造

| 段階 | 観察される特徴 |
|---|---|
| **1 不十分** | 「リファクタすべき」「より良く」レベル、構造化なし |
| **2 改善余地** | 提案はあるが具体性欠如、優先順位なし、長文フラット |
| **3 達成** | 提案ごとに具体的な実装方針、優先順位表あり、スキャナブル構造（見出し / 表） |
| **4 卓越** | 工数見積もり、選択肢の提示、**最小工数 × 高効果** の順序提案、即着手可能 |

**観察ポイント**:
- エグゼクティブサマリーで 30 秒で全体把握できるか
- 提案が「次に何をするか」のレベルまで具体化されているか
- 大規模工数と小規模工数が分離され、すぐ着手できる項目が独立しているか

---

### 軸 6: 客観性

| 段階 | 観察される特徴 |
|---|---|
| **1 不十分** | 否定的のみ、良い点ゼロ、確認なしで断言（後で「仕様でした」となる） |
| **2 改善余地** | 表面的な賞賛 +  否定中心、不確実性の表明なし |
| **3 達成** | 良い点を**具体的に**挙げる、不確実性は明示、断定を避ける |
| **4 卓越** | **誤検出を率直に訂正する独立セクション**、レビュー者自身の限界を明示、対立解釈を併記 |

**観察ポイント**:
- 「良い」「優秀」が抽象的賛辞ではなく、具体的な箇所・パターンに対して向けられているか
- 確信度の異なる主張が**同じ語気**で書かれていないか
- レビュー後に「実はそうじゃなかった」項目が出たときに訂正する仕組みがあるか

---

## 3. レッドフラグ（該当時は 1 段階以上下げる）

以下のいずれかが見られた場合、他軸が良くても全体評価を下げる。

1. **確認なしの「CRITICAL」断言** が実は仕様だった（特に検証可能だった項目）
2. **file:line がゼロ** の主張が全体の過半
3. **全項目が同じ severity**（全 HIGH 等）
4. **プロダクト哲学を真っ向から否定**するが、その否定の根拠が無い
5. **修正案ゼロ**（「やり直すべき」のみ）
6. **良い点の言及ゼロ**（10,000 行以上のコードベースで何も良くないことは無い）
7. **盲点に無自覚**（「全部見た」と主張するが、明らかな未調査領域がある）

## 4. 採点シート

```
軸 1. エビデンス品質              ___ / 4
軸 2. 重大度キャリブレーション      ___ / 4
軸 3. 網羅性                       ___ / 4
軸 4. コンテキスト適合              ___ / 4
軸 5. 実行可能性 & 構造             ___ / 4
軸 6. 客観性                       ___ / 4
                              合計 ___ / 24

レッドフラグ該当数: ___ 件
最終調整後スコア: ___ / 24
```

### 総合判定

| スコア | 判定 | 意味 |
|---|---|---|
| 22-24 | **A** | 出荷品質。即実装着手可能 |
| 18-21 | **B** | 土台は信頼できる。一部追加検証後に着手 |
| 14-17 | **C** | 大幅な再検証 / 補強が必要 |
| 13 以下 | **D** | やり直し推奨。判断に使えない |

**重要な注記**: 軸 1（エビデンス品質）と軸 6（客観性）が 2 以下の場合、他軸が良くても**全体を 1 段階下げる**。これらは報告書の信頼性の基礎であり、低いと他軸の判定そのものが疑わしくなるため。

## 5. 自己評価例: 今回の `2026-05-28-code-review.md` を採点

参考として、本ルーブリックを 2026-05-28 のレビュー報告書に適用した結果。

| 軸 | 段階 | 根拠 |
|---|---|---|
| 1. エビデンス品質 | **4** | 直接検証 ✅ / エージェント報告 🔍 / 訂正 ⚠️ の凡例を冒頭に置き、各指摘の検証ステータスを区別。付録 A に直接確認した対象一覧 |
| 2. 重大度キャリブレーション | **3** | P0〜P3 + 統合優先順位表（工数列付き）。多次元評価はあるが、影響 × 発生確率の数値化までは未到達で 4 ではなく 3 |
| 3. 網羅性 | **4** | 11 領域カバー（cross-process / AI / CloudKit / Widget / 拡張）+ アプリ固有の Agentic loop / LintEngine まで踏み込み。付録 B で規模メトリクス開示 |
| 4. コンテキスト適合 | **3** | CLAUDE.md / DESIGN.md / spec 番号を参照、"calm UX" 哲学を尊重した提案（能動操作だけ surface 等）。完全 4 にするには既存設計判断の背景説明をさらに深掘りする余地 |
| 5. 実行可能性 & 構造 | **4** | エグゼクティブサマリー / 統合優先順位表（工数付き）/ 即着手可能項目を明示 / 章番号で navigable |
| 6. 客観性 | **4** | 第 1 章を **「訂正事項」** に充てて Safari `"application.id"` の誤検出を率直に訂正。良い点（fetch 防御の堅牢さ、デザインシステムの一貫性、ハルシネーション対策）を具体的に列挙 |

**合計**: 22 / 24 → **A**
**レッドフラグ**: 該当なし（CRITICAL 誤検出は第 1 章で先回り訂正済）
**最終調整後**: 22 / 24 → **A**

### 自己評価からの教訓
- 軸 2 と 4 を 4 に上げるには、「修正コスト × 影響 × 発生確率」の半定量化（簡単な行列でも）、および既存設計判断（spec で計画された経緯）への踏み込みが必要。
- 自己評価は楽観的に振れがちなので、第三者（チームメイト / 別 AI）に同じルーブリックで採点してもらうのが理想。

## 6. 使い方のヒント

- **作成時**: 着手前にルーブリックを開き、各軸で 3 を狙う最小限の構造（凡例、優先順位表、訂正セクション、良い点列挙）を最初から組み込む。
- **採点時**: 軸ごとに付箋を作って読み進め、観察ポイントごとに具体例を 1 つ拾う。具体例が拾えない軸は 2 以下。
- **複数報告書の比較**: 同じプロジェクトに対する複数の報告書を採点して、強み / 盲点を補完。AI 生成報告書は軸 3（網羅性）は得意だが軸 4（コンテキスト適合）が落ちやすいので、人間レビューでそこを補強。
- **改訂運用**: 報告書 v1 → 採点 → 弱い軸を補強した v2、というイテレーション。

## 付録: アンチパターン早見表

| アンチパターン | 該当軸 | 対処 |
|---|---|---|
| 「これは CRITICAL バグ」が仕様 | 1, 6 | 検証前の断定を禁止、訂正セクションを設ける |
| 全部 HIGH / すぐ対応 | 2 | 影響と工数の二軸で優先度を分離 |
| UI だけ深掘りしてセキュリティ未調査 | 3 | チェックリスト方式で領域漏れを防ぐ |
| 「業界標準」を絶対視 | 4 | プロダクト哲学を最初に確認、その中で提案 |
| 「リファクタすべき」で終わる | 5 | 具体的な実装方針 + 工数見積もりを必須化 |
| 良い点ゼロ | 6 | 「優秀な実装」を最低 3 つ挙げる枠を強制 |
| 「全部見た」と主張するが明らかな盲点 | 3, 6 | 未調査領域を付録で明示 |
