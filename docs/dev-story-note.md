# 100個の「spec」でiPhoneアプリを作った話 — AIと一緒に、端末内で完結する第二の脳を育てるまで

> このアプリ「Knowledge Base」は、保存した記事をAIが端末の中だけで整理し、自分専用の百科事典（Wiki）に編んでいくiOSアプリです。個人開発で、100を超える「spec（仕様）」を積み重ねて作りました。この記事は、その開発の一部始終 — どう始め、どんな手法で、何につまずき、設計思想がどう変わっていったか — の記録です。

> 🖼️ **【図①｜ヒーロー】** 完成したアプリの「ナレッジ」タブ（育った状態）のスクショ、または `specs/` フォルダに100個以上のspecが並んでいるエディタの画面。「個人で100スペック作った」が一目で伝わる方を。
>
> ℹ️ この記事は `🖼️【図◯】` の箇所に画像（スクショ・図・ログ）を差し込む前提です。開発記事なので、アプリ画面より**実際のログや図のほうが刺さります**。詳細は末尾の「図版の入れどころ」を参照。

## はじめに：なぜ作ったのか

「あとで読む」に放り込んだ記事を、二度と見返さない。タブは開きっぱなし、ブックマークは墓場。情報を**保存する**道具は世の中に溢れているのに、大半は保管庫で終わる。

問題は保存ではなく、**その後**にありました。

- 保存した知識がバラバラのままで、全体像が見えない
- 同じテーマの記事が増えても、自分の理解が更新されない
- 「あの記事どこだっけ？」と探す時間ばかり積み上がる

だったら「保存の後」をぜんぶAIに任せよう。**ユーザーは保存するだけ**。AIが読んで、要点を抜き、テーマごとにページを編み、新しい記事が来たら既存の理解を更新する。人は整理しない。AIが整理して、人は必要なときに見て、間違いがあれば直すだけ。

それが Knowledge Base の出発点でした。

## 開発手法：「仕様駆動 × AIペアプロ」

このアプリの一番の特徴は、機能そのものより**作り方**かもしれません。

### Spec Kit による仕様駆動開発

すべての機能を、番号付きの「**spec（仕様）**」として、決まったパイプラインで進めました。

```
/specify   → spec.md   何を・なぜ作るか、ユーザーストーリー、受け入れ条件
/plan      → plan.md   設計・データモデル・契約・「憲法チェック」
/tasks     → tasks.md  依存順に並んだ、テスト可能なタスク
/implement → コード + ユニットテスト（ビルド成功・テスト通過が条件）
```

思いつきでコードを書き始めるのではなく、まず「何を・なぜ」を文章にし、設計を詰め、タスクに割って、初めて実装する。地味ですが、これが100スペックを破綻させずに積み上げられた最大の理由でした。

### プロジェクト「憲法」

`constitution.md` という7つの核となる原則を置き、**すべての設計をこれに照らしてチェック**しました。

1. プライバシーファースト・ローカルファースト
2. MVPファースト開発
3. ソースに基づいた知識生成（根拠なし出力を出さない）
4. iOSの実現可能性を重視する
5. シンプルで落ち着いたUX（不安を煽らない）
6. 保守しやすいSwiftUIアーキテクチャ
7. 日本語ファースト

新機能を足したくなるたび、「これは憲法に反していないか？」と立ち止まる。後述する大転換も、この原則があったからこそ踏み切れました。

### AIとのペアプログラミング

実装は Claude Code（Opus）と二人三脚で進めました。仕様を渡し、設計を相談し、コードを書き、テストを通し、実機ログを一緒に読んで直す。AIが「作業者」、私が「方向を決める人」。この記事自体も、その対話の記録から生まれています。

### 技術スタック

- **Swift 6 / SwiftUI / SwiftData + CloudKit**
- **Apple Foundation Models**（端末内LLM、`@Generable`で構造化出力）
- NaturalLanguage（埋め込み・言語判定）/ Translation / Speech（文字起こし）/ Vision（OCR）/ PDFKit / Accelerate
- BGTaskScheduler（バックグラウンド処理）、Share Extension、Safari Web Extension、ウィジェット

すべての知能を**クラウドではなく端末の中**で動かす。これが技術的にも思想的にも、一番こだわった制約でした。

## spec 001〜101：開発の流れと、その変化

100を超えるスペックは、大きく7つの局面に分かれます。

### 第1局面（spec 001-010）：土台を作る

まずは「記事を保存して、AIが要約する」という核を作りました。Share Sheetからの保存、本文抽出、AIによる要約。ここで早くも最大の敵に出会います — **Apple Foundation Modelsの4096トークンという狭い窓**。長い記事は一度に処理できない。そこで本文を分割して順に要約し、それをさらに階層的にまとめ直す「チャンク要約」「階層要約」を実装。バックグラウンドで少しずつ進める仕組みも入れました。

### 第2局面（spec 011-018）：知識化とUI

抽出した知識を「見せる」段階。AIブレインタブ、自動タグ付け、既存記事への一括タグ付け（backfill）、統一デザインシステム、カテゴリ分類、ダークモード対応。そして「知識Clip」タブ — カテゴリごとにAIがダイジェストを作る画面。この頃から「ただの保存アプリ」から「知識が育つアプリ」へと色が出てきます。

### 第3局面（spec 019-034）：取り込みの拡張とAIチャット

ここで最初の**手痛い撤退**がありました。**spec 019：Chrome連携の撤回**。iOSのChromeはx-callback-urlで「今開いているタブのURL」を返してくれない — 技術的に実用化不可能と判明。憲法の「iOSの実現可能性を重視する」に従い、潔く撤回しました。代わりにSafari Web Extension、そして**AIチャット（RAG）** を実装。保存した記事だけを根拠に、出典付きで質問に答える。PDF対応も入りました。

### 第4局面（spec 035-050）：VISION v1 — 4つの知能

初めて明確な「プロダクトビジョン」を、17問の自己対話で固めました。「最近のあなた」差分ダイジェスト、動的トピック自動発見（K-means）、時系列の事実上書き提案、概念ページ（ConceptPage）、AIチャットの答えの永続化、そして「理解度」を育てる家庭教師ループ。野心的に機能を積み上げた時期です。…が、これが後の反省につながります。

### 第5局面（spec 051-061）：iCloud同期とV3再設計

**spec 051：iCloud同期**。SwiftDataのCloudKit対応は地獄でした。14個のリレーションをすべてOptional化し、100箇所近いアクセスを安全化する、88ファイル・3000行規模の改修。一方で**spec 056：V3.0再設計**では、増えすぎたタブを4→3に削減。足し算で膨らんだUIを、初めて本気で引き算しました。

### 第6局面（spec 063-088）：大転換 — LLM Wiki

そして、開発全体で最大の転換点が訪れます。

V3.0まで「**AIが裏で多層的に全部自動整理する**」設計を積み上げた結果、2つの問題が深刻化していました。

- **重い**：1記事の保存でAIを最大12〜15回呼ぶ（知識抽出＋矛盾検出10回＋グラフ＋概念合成…）。トークン超過で常にフォールバック。起動時の一括処理が100記事で3〜5分。
- **情報過多**：「人物・モノ・概念」を表すデータモデルが**7個に分裂**（ConceptPage / GraphNode / UserTopic / KnowledgeEntity / KnowledgeDigest…）。しかも大半はユーザーから見えていない。

根本原因は「全部をAIが多層的に自動整理する」という設計そのものでした。

ここで効いたのが、**Andrej Karpathy の「LLM Wiki」** という考え方です。

> *「You can outsource your thinking, but you cannot outsource your understanding.（思考は外注できても、理解は外注できない）」*

7つの分裂した概念を「**Wikiページ**」という1つに畳む。引き算ではなく「**統合による引き算**」。これで重さ（生成系統を1本に）と情報過多（概念を1種類に）を同時に解消する。

> 🖼️ **【図②｜大転換】** 「7つの分裂モデル → 1つのWikiページ」を表す**Before/Afterの概念図**（手描きでもOK）。記事のハイライトなので、ここは図を自作する価値あり。Karpathyの3層（元資料／Wiki／ルール）を並べた図でも。

VISION v2として「**読んだ記事をAIが裏で1つの百科事典に編さんし続け、開くだけで自分だけの知識が育っていくのが見える、優しい第二の脳**」を一文ビジョンに据え直し、設計を大改造しました。Karpathyの3層（不変の元資料 / AIが書くWiki / 運用ルール）をそのままアプリの構造に写し取り、「軽さ優先」を最重要原則に格上げ。1記事あたりのAI呼び出しを2〜3回に削り、見えない裏処理（グラフ生成・トピック分類）を止めました。News+風のフィードもこの局面です。

### 第7局面（spec 089-101）：リリース準備と品質の作り込み

最後は出荷に向けた仕上げ。要点ごとの出典表示、**iKnow → 「Knowledge Base」へのリブランド**、AIチャットの会話型RAGチューニング（番号引用・検索優先・履歴考慮）、プライバシーマニフェスト・暗号輸出申告などのリリース必須対応。そして「あらゆる入力の取り込み」 — テキスト・PDF・**写真OCR・音声の文字起こし**まで、共有シートひとつで知識化できるように拡張。さらにカテゴリ分類を**自己改善ループ**化し、最後に多言語記事の翻訳を安定化させました。

## 一番の敵：4096トークンとの終わらない戦い

この開発を貫く技術的な主題は、ずっと「**トークン窓4096との戦い**」でした。

端末内のAIは速くて無料でプライベートですが、一度に扱える文字数が小さい。しかも厄介なことに、`@Generable`で「こういう構造で出力して」と指示すると、その**出力予約**だけで窓をかなり食う。本文を削っても、出力スキーマが大きいと溢れる。

溢れると何が起きるか。AIがまともなJSONを返せず、貧弱な「essenceの羅列」にフォールバックして、知識の質が落ちる。実機ログに何度も `4091 tokens` の文字が並び、そのたびに対策を打ちました。

- 出力スキーマをスリム化する
- 溢れたら、より小さいスキーマで1回だけ再試行する（adaptive retry）
- 暴走を防ぐハード上限（`maximumResponseTokens`）を入れる
- そもそもトークンを**実測**するプローブ（TokenProbe）を仕込み、各スキーマが窓の何%を食うか可視化する

「入力を削りすぎると質が落ちる、削らないと溢れる」というジレンマと、最後までずっと付き合いました。

> 🖼️ **【図③｜トークン実測】** TokenProbeの実機ログ（`schema ExtractedKnowledgeOutput = 552 tokens (窓の13%)` のような行が並ぶ画面）のスクショ。**生の数字**が説得力を出すので、コンソールログをそのまま貼ると良い。

## 賢くなる仕組み：間違いを学習するAI

端末内モデルが一番ぶれるのはカテゴリ分類でした。人名を「スポーツ」に入れたり、存在しないカテゴリを作ったり。そこで、**使うほど良くなるループ**として作り直しました（spec 097）。

```
① 取り込み時に分類して、確信度［High / Medium / Low］を付ける
     Low → 保守的に「その他」へ ／ Medium・その他 → 「要確認」
② 裏の整理ループが、不確実なものから優先的に再分類
③ ユーザーが「分類の確認」画面で正しい分野に直す
     → その修正を「正解例」として端末内に記録
④ 次回の分類で、その正解例を手本（few-shot）としてAIに渡す
     → 同じ間違いを繰り返さない
```

直すほど自分のデータに馴染んで賢くなる。**人間の確認がAIの教師になる**。「理解は外注できない」という思想が、そのまま実装に現れた部分です。

## 開発中に起きた、リアルな事件たち

きれいな話ばかりではありません。実機ログと格闘した、生々しい記録も残しておきます。

- **「This store went missing」クラッシュ**：本文の見直し機能で、起動時の抽出処理が書き込んでいる最中にデータを消してしまった。「二度手間になるなら、見直し依頼が来たら今の抽出を止めて本文を先に直すべきでは？」というユーザーの指摘で、設計ごと作り直し。
- **見直し完了のポップアップが出ない**：Sceneのbody直書きでは@Observableの変化で再評価されない、というSwiftUIの罠。専用Viewに切り出して解決。
- **2つ目のCloudKitコンテナが落ちる**：分類の学習データを別ストアでiCloud同期しようとしたら、メインストアと同じiCloudコンテナを二重ミラーリングして競合し、実機で「Store Removed」。学習ストアはローカル専用に変更して安定化。
- **翻訳でtranslationdが27回クラッシュ**：英語の巨大論文を30チャンクに割ったら、参照や数式の断片が「インドネシア語」「フランス語」に誤検知され、無駄な翻訳を繰り返してクラッシュ連発、1記事に6.3分。言語判定を**記事単位で1回**に寄せて解決。
- **マージの取りこぼし**：PRをマージした後に追加コミットをpushしたら、すでにマージ済みで取りこぼされる、というGitの事故が2回。cherry-pickで復旧しつつ、「追加分は新しいPRにする」という運用に。

どれも、**実機で動かして初めて見える**問題でした。シミュレータでは端末内AIも翻訳も本来の姿で動かない。「最後は実機」という当たり前を、何度も思い知らされました。

> 🖼️ **【図④｜事件のログ】** `translationd ... crashed` が大量に並ぶログや、`This store went missing` のクラッシュログのスクショ。**生々しい失敗の証拠**は開発記事で一番読まれる部分。1枚あると一気にリアルになる。

## 振り返り：この開発で学んだこと

### 1. 引き算は足し算より難しい

機能を足すのは簡単。一番の設計判断は、いつも「**何を作らないか**」でした。「Wikiに質問してページ化」は作らない（操作を増やさない）。見えない裏処理は止める。概念モデルは7種類を1つに畳む。「開くだけで価値が出る」一点に絞り込むことが、結局いちばん効きました。

### 2. ビジョンは途中で書き直していい

VISIONは2回書き直しました。最初は4つの知能を野心的に積んで、重さと情報過多に行き詰まり、LLM Wikiで畳み直した。**走りながら方向を修正する**ことを、ドキュメントとして堂々と残す。これが個人開発の強みだと思います。

### 3. 制約を中心に据えると、設計が決まる

「端末内AIだけで作る」「4096トークン」「プライバシー優先」。これらは不便な制約ですが、制約があるからこそ「軽さ優先」「ソースに辿れる」「見られないものは生成しない」という設計が一意に決まっていきました。

### 4. 仕様駆動 × AIは、個人開発と相性がいい

100スペックを一人で破綻させずに回せたのは、毎回「spec → plan → tasks → implement」を踏み、憲法でブレを抑え、AIに実装を任せて自分は方向に集中したからでした。

## これから

方向性は一貫しています — **機能を足すより、「開くだけで自分の知識が育っている」体験を磨く**。

端末間での学習同期、概念ページ同士の関係発見の深化、「今週のハイライト」ダイジェスト。やりたいことはまだありますが、軸はぶらさずにいきます。

読んだものが、勝手に、自分だけの百科事典になっていく。そんな静かな第二の脳を、これからも端末の中で育てていきます。

### 図を足すなら（あると強い）
- **spec数の推移グラフ**（spec番号 × 時系列、局面ごとに色分け）— 「積み上げ」が視覚化できる
- **開発フロー図**：`/specify → /plan → /tasks → /implement` の4ステップ図

- **AIとの会話のスクショ**：実際にClaude Codeと設計を相談している画面（“AIペアプロ”の実物）

### コツ
- ログのスクショは**等幅フォント・ダーク背景**のまま貼ると“本物感”が出る。
- 図は無理に凝らず、**手描き → 写真**でも十分伝わる（むしろ個人開発記事では好まれる）。
- アプリ画面を使う場合は、別記事「アプリ紹介note」のスクショ撮影チェックリストと同じものを流用できる。
