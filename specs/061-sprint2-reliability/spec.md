# Feature Specification: Sprint 2 信頼性改善 4 件

**Feature Branch**: `061-sprint2-reliability`
**Created**: 2026-05-30
**Status**: Draft
**Input**: code review FINAL report の Sprint 2 = 信頼性 P1 から 4 件 (P1-2 / P1-3 / P1-6 / P1-7)

## 背景

spec 059 (PR #18) / spec 060 (PR #19) で出荷ブロッカーと token 超過を解消後、code review FINAL report の信頼性 P1 から、別 spec 確定済の P1-4 (LintEngine staging) と完了済 P1-10 を除いた 4 件をまとめて解消する。全 4 件、実コードで root cause verified 済。

**1 文の本質**: 「黙って失敗する・弾き返す・クラッシュする・起動が遅い — ユーザーの信頼を削る 4 つの綻びを塞ぐ」

## User Scenarios & Testing *(mandatory)*

### User Story 1 - iCloud Toggle のバウンス解消 (Priority: P1)

Settings で iCloud 同期トグルをタップすると、確認ダイアログが出る前にスイッチがいったん元の位置に弾き返ってから、確認 OK でやっと反映される。ユーザーには「トグルが効かない / 壊れている」と見える。タップ後に位置がぶれず、確認 → 反映が自然に繋がるようにする。

**Why this priority**: 既存ユーザーが設定で必ず触る箇所。動作が壊れて見えると iCloud 同期そのものへの不信に直結する。

**Independent Test**: iCloud トグルをタップ → 確認ダイアログ表示中もスイッチがバウンスしない → キャンセルで元位置 / OK で反転、を確認。

**Acceptance Scenarios**:

1. **Given** iCloud 同期 OFF, **When** トグルをタップ, **Then** スイッチが弾き返らずに確認ダイアログが出る
2. **Given** 確認ダイアログ表示中, **When** キャンセル, **Then** スイッチは元の位置 (OFF) に戻る
3. **Given** 確認ダイアログ表示中, **When** OK, **Then** スイッチが ON になり再起動案内が出る (既存挙動維持)

---

### User Story 2 - ユーザー操作のサイレント失敗を可視化 (Priority: P1)

チャット履歴の削除・タグの追加/削除・回答のピン/既読/削除・概念ページのフォロー切替といったユーザー操作は、内部で保存に失敗しても画面上は成功したように見え、実際の保存内容と食い違う。失敗を黙殺せず、記録 (ログ) を残し、削除のような取り返しのつかない操作では失敗をユーザーに伝える。

**Why this priority**: 「やったはずの操作が実は反映されていない」は信頼を最も損なう。CloudKit 同期競合や移行エラーで実際に起こり得る。

**Independent Test**: 保存が失敗する状況を作り、操作後に失敗が記録され、削除操作では失敗が画面に伝わることを確認。

**Acceptance Scenarios**:

1. **Given** 保存が失敗する状況, **When** チャット履歴削除などの操作を行う, **Then** 失敗がログに記録される (黙殺されない)
2. **Given** 保存が失敗する状況, **When** 削除のような破壊的操作を行う, **Then** 失敗がユーザーに伝わる (成功表示にならない)
3. **Given** 保存が成功する通常状況, **When** 各操作を行う, **Then** 従来通り正常に反映される (退行なし)

---

### User Story 3 - 起動時クラッシュの回避 (Priority: P1)

端末内のデータ保存領域 (ローカルストア) の作成に失敗すると、現状はアプリが即クラッシュし、ユーザーは再インストール (= 全データ消失) 以外に復旧手段がない。失敗してもアプリは起動し、状態をユーザーに伝える (再インストールを強制しない)。

**Why this priority**: データ中心アプリでの hard-crash は最悪のユーザー体験。稀だが起きると蓄積した知識ベースを失わせる。

**Independent Test**: ローカルストア作成が失敗する状況で、アプリがクラッシュせず起動し、状態が表示されることを確認。

**Acceptance Scenarios**:

1. **Given** ローカルストア作成が失敗する状況, **When** アプリを起動, **Then** クラッシュせずに起動し、データ読み込み失敗の状態が伝わる
2. **Given** ローカルストアが正常な通常状況, **When** アプリを起動, **Then** 従来通り全機能が動作する (退行なし)

---

### User Story 4 - 起動時 backfill の並列化 (Priority: P1)

アプリ起動時、AI による整理処理 (本文抽出・知識抽出・タグ付け・分類・ダイジェスト・埋め込み・トピック・概念ページ等) が一列に順番待ちで実行され、互いに独立な処理まで直列になっている。独立な処理を同時進行させ、起動から初期表示までの待ち時間を短縮する。

**Why this priority**: cold start の「何も起きない時間」を縮め、起動直後の体感を改善する。

**Independent Test**: 多数の記事がある状態で起動し、独立 backfill が同時進行して全処理が完了し、既存機能に退行がないことを確認。

**Acceptance Scenarios**:

1. **Given** 多数の未処理記事がある, **When** アプリを起動, **Then** 依存関係のない整理処理が同時進行し、全処理が完了する
2. **Given** 依存関係のある処理 (本文抽出 → 知識抽出), **When** 起動 backfill が走る, **Then** 順序が保たれる (依存違反なし)

---

### Edge Cases

- **P1-2**: トグルを連続タップ / 確認ダイアログを途中で外す → 状態が pending のまま矛盾しない。
- **P1-3**: 裏側の自動処理 (backfill / 再生成) の失敗は従来通り黙ってリトライ対象 (calm UX、本 spec の surface 対象外)。
- **P1-6**: ローカルストア失敗時の代替起動ではデータ消失自体は防げない場合がある (crash 回避が目的)。debug ビルドでは原因に気付ける形にする。
- **P1-7**: 並列化した backfill のいずれかが失敗しても、他の処理と起動完了をブロックしない。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: iCloud 同期トグルのタップ時、確認前にスイッチ位置がバウンスしてはならない (P1-2)。
- **FR-002**: iCloud 同期トグルは、確認のキャンセルで元の状態に戻り、確認の承認で新しい状態に反映されなければならない (P1-2)。
- **FR-003**: iCloud トグルの確認ダイアログ・再起動案内など既存挙動を変更してはならない (P1-2)。
- **FR-004**: ユーザーの能動操作 (チャット履歴削除 / セッション削除 / タグ追加・削除 / 回答ピン・既読・削除 / 概念ページフォロー切替) が失敗したとき、失敗を記録 (ログ) しなければならない (黙殺禁止) (P1-3)。
- **FR-005**: 破壊的操作 (削除) が失敗したとき、ユーザーに失敗が伝わらなければならない (成功表示にしてはならない) (P1-3)。
- **FR-006**: 裏側の自動処理 (backfill / 再集約等) の失敗の扱いは変更してはならない (calm UX 維持、P1-3 surface 対象外)。
- **FR-007**: ローカルストア作成に失敗してもアプリはクラッシュしてはならず、起動して状態をユーザーに伝えなければならない (P1-6)。
- **FR-008**: ローカルストアが正常な通常状況では、全機能が従来通り動作しなければならない (P1-6 退行なし)。
- **FR-009**: 起動時の整理処理のうち、依存関係のない処理は同時進行できなければならない (P1-7)。
- **FR-010**: 依存関係のある起動処理 (本文抽出 → 知識抽出の連鎖) の順序は保たれなければならない (P1-7)。
- **FR-011**: 本修正は永続化スキーマ (@Model) を変更してはならない。

### Key Entities

本 spec はデータモデルを変更しない。関与する既存概念:

- **ローカルストア**: 端末内のデータ保存領域。作成失敗時の挙動を変える (crash 回避)。
- **エラー記録**: ユーザー操作失敗の記録手段 (端末内ログ)。新規の軽量な記録の仕組み。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: iCloud トグルをタップしてもスイッチが弾き返らず、キャンセルで元位置・承認で反転する。
- **SC-002**: ユーザー操作失敗時にログが記録され、削除操作の失敗がユーザーに伝わる。
- **SC-003**: ローカルストア作成失敗時にアプリがクラッシュせず起動する。
- **SC-004**: 起動時に依存関係のない整理処理が同時進行し、全処理が完了する (依存順序違反ゼロ)。
- **SC-005**: クリーンビルドが成功し、本 spec 由来の警告が 0 件。
- **SC-006**: 既存 unit test が全て regression なく PASS する (エラー記録のテストを含む)。

## Assumptions

- エラー記録は端末内のシステムログ (外部送信なし) を用いる軽量な仕組みとする。
- 破壊的操作 (削除) のみ失敗を画面に伝え、トグル・ピンなど非破壊操作は記録 + 状態復元で十分とする (calm UX、強い警告ダイアログを濫用しない)。
- ローカルストア失敗時の代替起動は「クラッシュ回避と状態通知」を最小目標とし、完全な復旧 UI (リトライ / ローカル限定切替の本格画面) は最小実装に留める。
- 起動処理の「並列化」は、端末内サービスが順次 await で待っていた独立処理を同時進行させることを指す (重い同期計算のバックグラウンド退避は別 spec)。
- 起動処理は全体としてメインの実行文脈で走るため、並列化は待ち時間の重ね合わせによる短縮であり、CPU 計算自体の並列化ではない。

- P1-9 チャット行単位のデータ取得最適化 (別 spec)。
