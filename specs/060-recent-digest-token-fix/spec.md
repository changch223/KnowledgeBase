# Feature Specification: RecentDigest token 超過修正 + SchemaLoader bundle 同梱

**Feature Branch**: `060-recent-digest-token-fix`
**Created**: 2026-05-30
**Status**: Draft
**Input**: code review FINAL report の P1-10 + 実機ログで顕在化した SchemaLoader bundle 漏れ

## 背景

spec 059 (PR #18) で P0 5 件を解消後、実機ログで 2 件の既存問題が顕在化した。両方とも軽い修正で、ユーザー体感に直結する。

1. **P1-10 (token 超過)**: 「最近の記事」AI ヘッドラインが毎回 token 上限超過で AI 生成に失敗し、簡易表示 (fallback) に落ちている。
2. **SchemaLoader bundle 漏れ**: AI 整理ロジックの指示書 (schema) がアプリに同梱されておらず、毎回コード内の予備設定で動いている (本来の外出し設計が機能していない)。

**1 文の本質**: 「token 超過で出ない『最近の記事』AI ヘッドラインを復活させ、schema 外出し設計を実際に機能させる」

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 「最近の記事」AI ヘッドラインの復活 (Priority: P1)

ユーザーが知識 Clip タブを開くと最上部に「最近の記事」セクションがある。本来 AI が複数記事を統合した 1 文ヘッドライン + 主要テーマ chips を生成して表示するはずが、内部で扱うデータ量が多すぎて AI 生成が毎回失敗し、簡易的な記事タイトル並べ (fallback) に落ちている。AI 生成が成功するようにし、本来の統合ヘッドラインを表示する。

**Why this priority**: 知識 Clip タブ最上部 = ユーザーが最も見るエリア。AI の価値 (複数記事の統合要約) が出ず、ただのタイトル列挙になっており、プロダクトの中核体験が損なわれている。

**Independent Test**: 記事を多数 (30 件以上) 保存した状態で知識 Clip タブを開き、「最近の記事」に AI が生成した自然な 1 文ヘッドライン + テーマ chips が表示され、内部ログに token 超過エラーが出ないことを確認。

**Acceptance Scenarios**:

1. **Given** 30 件以上の記事が保存されている, **When** 知識 Clip タブを開く, **Then** 「最近の記事」に AI 生成の統合ヘッドライン (1 文) + テーマ chips が表示される
2. **Given** AI 生成処理が走る, **When** ヘッドラインを生成する, **Then** 内部で扱う情報量が AI の処理上限を超えず、生成が成功する (簡易 fallback に落ちない)
3. **Given** AI が利用不可な端末 (Apple Intelligence 未対応), **When** 知識 Clip タブを開く, **Then** 従来通り essence ベースの簡易ヘッドラインが表示される (既存 fallback 挙動を壊さない)

---

### User Story 2 - schema 外出し設計を実際に機能させる (Priority: P1)

AI が裏で行う整理ロジック (統合 / 矛盾解消 / 健全性判定の指示書) は、開発者が調整しやすいよう外部ファイル (schema) に切り出す設計になっている。しかしその schema ファイルがアプリに同梱されておらず、起動時に毎回「見つからない」ためコード内の予備設定で動作している。schema ファイルをアプリに同梱し、設計通り schema 由来で動作するようにする。

**Why this priority**: ユーザーには直接見えないが、AI 整理ロジックの調整 (チューニング / AB test) の土台。現状は設計が形骸化しており、schema を編集しても反映されない。

**Independent Test**: アプリ起動時の内部ログに「schema をアプリ内蔵ファイルから読み込んだ」旨が出て、「見つからない (予備設定使用)」が出ないことを確認。

**Acceptance Scenarios**:

1. **Given** アプリを起動する, **When** schema 読み込み処理が走る, **Then** アプリ同梱の schema ファイルから読み込みに成功する
2. **Given** schema ファイルが何らかの理由で読めない, **When** 読み込み処理が走る, **Then** 従来通りコード内予備設定で安全に動作する (二重安全、既存 fallback 維持)

---

### Edge Cases

- **P1-10**: 記事が 0〜数件しか無い場合も AI 生成が成功する (件数下限でも動作)。
- **P1-10**: 1 記事の要約が異常に長い場合でも、全体が処理上限を超えないよう抑制される。
- **P1-10**: AI 利用不可端末では既存の簡易 fallback がそのまま動く (本修正で壊さない)。
- **SchemaLoader**: schema ファイル同梱後も、読み込み失敗時のコード内予備設定への fallback 経路は維持される (production 安全)。
- **SchemaLoader**: 単体テスト環境ではアプリ本体とは別のバンドルになるため、テスト方針は fallback 経路の検証を中心とする。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 「最近の記事」AI ヘッドライン生成に渡す情報量は、AI の処理上限 (コンテキストウィンドウ) を超えてはならない (P1-10)。
- **FR-002**: AI ヘッドライン生成に使う記事は、ヘッドライン + テーマ生成に十分な代表的な件数に制限されなければならない (全保存記事を列挙しない) (P1-10)。
- **FR-003**: AI ヘッドライン生成が成功する場合、簡易 fallback ではなく AI 生成結果が表示されなければならない (P1-10)。
- **FR-004**: AI が利用不可な端末では、従来の essence ベース簡易ヘッドラインが引き続き表示されなければならない (既存 fallback 維持) (P1-10)。
- **FR-005**: 記事数が多い場合の差分判定・件数表示など、ヘッドライン生成以外の既存挙動は変更してはならない (P1-10)。
- **FR-006**: AI 整理ロジックの schema ファイルはアプリに同梱され、起動時にアプリ内蔵ファイルから読み込めなければならない (SchemaLoader)。
- **FR-007**: schema ファイルの読み込みに失敗した場合、コード内の予備設定で安全に動作しなければならない (既存 fallback 維持、二重安全) (SchemaLoader)。
- **FR-008**: 本修正は永続化スキーマ (@Model) を変更してはならない。

### Key Entities

本 spec はデータモデルを変更しない。関与する既存概念:

- **記事 (Article)**: AI ヘッドライン生成の入力。参照のみ、変更なし。
- **schema 指示書**: AI 整理ロジック (統合 / 矛盾 / 健全性) の外部設定ファイル。アプリに同梱する。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 30 件以上の記事が保存された状態でも、AI ヘッドライン生成に渡す情報量が処理上限内に収まり、生成が成功する。
- **SC-002**: 実機で「最近の記事」に AI 生成ヘッドラインが表示され、内部ログに「最近の記事」由来の token 超過エラーが出ない。
- **SC-003**: アプリ起動時の内部ログに「schema をアプリ内蔵ファイルから読み込んだ」が出て、「見つからない (予備設定使用)」が出ない。
- **SC-004**: クリーンビルドが成功し、本 spec 由来の警告が 0 件。
- **SC-005**: 既存 unit test が全て regression なく PASS する (ヘッドライン生成 + schema 読み込みのテスト含む)。

## Assumptions

- AI の処理上限 (コンテキストウィンドウ) は約 4096 トークン。日本語はおおむね 1 文字 ≈ 1 トークンに近いため、文字数ベースの概算で安全マージン (例: 3500 文字以内) を取れば実用上十分に超過を防げる。
- ヘッドライン + テーマ 3 個の生成には、最新の代表的な記事 (8〜10 件程度) で十分。全保存記事 (最大 30 件) を列挙する必要はない。
- schema ファイルはアプリ同梱用に専用の場所 (アプリリソース) に配置する。人間用ドキュメントとしての元ファイルは残してよい。
- 単体テストはアプリ本体とは別バンドルで動くため、schema 同梱の検証は中心的には fallback 経路で行い、実バンドル読み込みは実機/起動ログで確認する。
- 危険操作の確認や既存の fallback 経路は本修正で削除しない。
