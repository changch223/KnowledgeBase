# Feature Specification: 記事保存 (Share Sheet 経由)

**Feature Branch**: `001-save-article`
**Created**: 2026-05-04
**Status**: Draft
**Input**: ユーザー説明: "iOS Share Sheet で受け取った URL を、タイトルと保存日時とともにローカル保存し、保存済み一覧から元記事をブラウザで再閲覧できる。本文抽出・要約・分類は本 spec のスコープ外。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 共有から保存して一覧で確認できる (Priority: P1)

ユーザーが Chrome / Safari / X などのアプリで読んでいる記事を「あとで見返したい」と思ったとき、共有ボタンから KnowledgeTree を選ぶだけで、その記事の URL とタイトルが自動的にローカル保存される。後でアプリを開くと、保存した記事がタイトル付きで一覧に並んでいる。

**Why this priority**: 本機能の中核。これが動かなければ KnowledgeTree は何も保存できない単なる空のアプリになる。Constitution の MVP スコープ「信頼性の高い記事保存」「ローカル保存」を直接実現する最小単位。

**Independent Test**: 任意のアプリの共有メニューから KnowledgeTree を選び、その後 KnowledgeTree を起動して、共有した記事のタイトルが一覧に表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが Safari で記事ページを開いている、**When** 共有ボタン → KnowledgeTree を選ぶ、**Then** アプリの共有受け取り UI が立ち上がり、保存処理が完了して自動で閉じる。
2. **Given** 上記で保存した直後、**When** ユーザーが KnowledgeTree アプリを開く、**Then** 一覧の最上段に保存した記事のタイトルと URL が表示される。
3. **Given** 一覧に複数の保存済み記事がある、**When** ユーザーがアプリを開く、**Then** 一覧は保存日時の新しい順に並ぶ。
4. **Given** Chrome で開いている記事から共有する、**When** 共有メニュー → KnowledgeTree、**Then** Safari の場合と同じく URL とタイトルが保存される。
5. **Given** ユーザーが過去に保存した記事と同じ URL を再度共有する、**When** 共有メニュー → KnowledgeTree、**Then** 共有受け取り UI は「既に保存済みです」と短く表示し、新規レコードは作成されず、既存レコードの保存日時 (savedAt) は変更されない。

---

### User Story 2 - 元記事をブラウザで再閲覧する (Priority: P2)

ユーザーが一覧から特定の記事を読み返したいとき、その行をタップすると iOS 内蔵のブラウザビューが開き、元記事を読める。アプリから離れることなく、また外部ブラウザのタブを汚染することなく、軽快に元記事に戻れる。

**Why this priority**: 保存だけでき再閲覧できないと、保存しても記事に戻れず本機能の価値が半減する。Constitution Principle III「ソースに基づいた知識生成」の最低ラインとして、ユーザーがいつでも元ソースを確認できる導線を提供する。

**Independent Test**: 一覧の任意の行をタップし、内蔵ブラウザビューが開いて元 URL のページが表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 一覧画面に保存済み記事が表示されている、**When** ユーザーが行をタップする、**Then** iOS 内蔵ブラウザビューが画面を覆い、その記事の元 URL がロードされる。
2. **Given** 内蔵ブラウザビューが開いている、**When** ユーザーが「完了」を押す、**Then** ビューが閉じ、一覧画面に戻る。

---

### User Story 3 - 不要な記事を削除する (Priority: P3)

ユーザーが「もう要らない」と判断した記事を、リストからスワイプ操作で削除できる。これによりリストは管理可能なサイズに保たれ、Constitution Principle V「シンプルで落ち着いた UX」が維持される (溜まり続ける未消化記事の不安を発生させない)。

**Why this priority**: MVP のコア体験 (P1, P2) には含まれないが、削除手段が無いとリストが無限に成長して使用感が損なわれる。基本的なリスト操作として優先度 P3 で必須。

**Independent Test**: 一覧の任意の行を左にスワイプして「削除」を押し、その行がリストから消え、再度アプリを起動しても消えたままであることを確認する。

**Acceptance Scenarios**:

1. **Given** 一覧画面に保存済み記事が表示されている、**When** ユーザーが行を左にスワイプして削除アクションを実行する、**Then** その行は即座にリストから消える。
2. **Given** 上記で削除した直後、**When** アプリを再起動する、**Then** 削除した記事は復活せず、一覧から消えたまま。

---

### Edge Cases

- **共有 payload に URL が含まれない**: テキストや画像だけが共有された場合、共有受け取り UI は「URL が見つかりません」と短く表示してそのまま閉じる (記事として保存しない)。
- **共有 payload に Title が含まれない**: タイトルが取れない場合、URL のホスト名 (例: `example.com`) を仮タイトルとして保存する。
- **同一 URL を複数回共有**: 重複検出が動作する。既存レコードはそのまま保持され (savedAt 変更なし)、新規レコードは作成されず、共有受け取り UI で「既に保存済みです」と短く表示してから閉じる。URL の比較は文字列としての完全一致で行う (正規化なし)。
- **non-HTTP/HTTPS の URL** (`file://`、`mailto:` 等): 共有受け取り UI は「対応していない URL です」と短く表示してそのまま閉じる。
- **保存処理中にストレージ書き込み失敗**: 共有受け取り UI は短いエラーメッセージを表示して終了する。アプリ本体側の一覧には影響しない。
- **空状態 (一件も保存していない)**: 一覧画面は空ではなく、日本語で「共有メニューから記事を追加してみよう」と落ち着いたメッセージを表示する (不安喚起しない)。
- **極端に長いタイトル**: 一覧では 2 行で打ち切り表示する。詳細表示は不要 (内蔵ブラウザビューで元ページを見れば良い)。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: アプリは iOS 標準の共有メニュー (Share Sheet) に「KnowledgeTree」として表示され、選択されると共有された URL を受け取れる。
- **FR-002**: 共有受け取り処理は、共有された URL とページタイトルを取得できる。
- **FR-003**: 共有受け取り処理は、URL・タイトル・保存日時 (savedAt) をローカルに永続化する。
- **FR-004**: アプリ本体は起動時、保存済み記事を保存日時の新しい順に並べて一覧表示する。
- **FR-005**: 一覧の各行は記事タイトル (主) と URL (副) を表示する。
- **FR-006**: ユーザーは一覧の行をタップすると、iOS 内蔵のブラウザビューで元 URL のページを開ける。
- **FR-007**: ユーザーは一覧の行を左にスワイプし、削除アクションでその記事を一覧と保存先から削除できる。
- **FR-008**: 共有受け取り処理が URL を取得できないペイロード (テキストのみ・画像のみ・非対応スキーム) を受け取った場合、保存せずに簡潔なエラーメッセージを表示して閉じる。
- **FR-009**: 共有された Title が空の場合、URL のホスト名を仮タイトルとして保存する。
- **FR-010**: 本機能 (保存・一覧・閲覧・削除) はネットワークアクセスを発生させない (Principle I)。内蔵ブラウザビューを開いた後の通信はユーザーが意図した OS 標準のブラウザ動作であり、本アプリの送信ではない。
- **FR-011**: すべての UI 文言 (共有受け取り UI・一覧・空状態・エラーメッセージ・スワイプアクション・内蔵ブラウザビューのナビゲーションタイトル) は日本語で表示される (Principle VII)。
- **FR-012**: 主要なインタラクティブ要素 (一覧行・削除アクション・内蔵ブラウザビューの完了ボタン) は安定したアクセシビリティ識別子を持つ (Quality Gate / Principle V のテスト容易性)。
- **FR-013**: 一覧画面は 0 件状態のとき、不安を喚起しない落ち着いたメッセージで「共有メニューから記事を追加してみよう」と案内する (Principle V)。
- **FR-014**: 共有受け取り処理は、保存前に既存記事との **URL 完全一致** による重複チェックを実施する。URL の正規化 (末尾スラッシュ除去・トラッキングパラメータ除去等) は MVP では行わない。
- **FR-015**: 重複が検出された場合、新規保存は行わず、共有受け取り UI に日本語で「既に保存済みです」と短く表示してから自動で閉じる。既存レコードの savedAt は変更しない。

### Key Entities *(include if feature involves data)*

- **Article**: 保存された 1 件の記事を表す。
  - 必須属性: 一意識別子、元記事 URL、タイトル、保存日時 (savedAt)。
  - 関係: 将来の enrichment spec で 本文 / description / OG image / カテゴリ / 要約 が追加される予定。それらの派生データは本 Article への非 optional 参照を持つ (Principle III)。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは任意の対応アプリ (Chrome / Safari 等) の記事ページから、共有ボタンタップ → KnowledgeTree 選択の **2 タップ** で記事を保存できる。
- **SC-002**: 保存操作完了から、アプリを開いて当該記事が一覧の最上段に見えるまで、**コールド起動でも 2 秒以内**。
- **SC-003**: 一覧画面は 100 件の保存済み記事を含むとき、ベースライン端末上で **60 fps を維持しスムーズにスクロール** できる。
- **SC-004**: 一覧の任意の行から 1 タップで内蔵ブラウザビューが **300 ms 以内** に開き、元記事を読める。
- **SC-005**: 本機能の保存・閲覧・削除の全フローを通じて、KnowledgeTree アプリ自身は **記事内容の取得・送信を一切行わない** (ネットワークパケットゼロ)。
- **SC-006**: 初めてアプリを起動するユーザーが、最初の記事を保存して一覧で確認するまでの所要時間が **平均 30 秒以内**。
- **SC-007**: 削除操作を実行してから一覧から該当行が消えるまでの視覚フィードバックが **100 ms 以内**。
- **SC-008**: アプリと共有受け取り UI の全 UI 文言が日本語で表示され、英語混在やローカライズ漏れが **0 件**。
- **SC-009**: 同一 URL を再共有した場合、共有受け取り UI で「既に保存済みです」のメッセージが **1 秒以内** に表示され、一覧の重複は **0 件**。

## Assumptions

- **対象 OS / 端末**: iOS 26+ / iPadOS 26+。本 spec の保存・閲覧機能は Apple Foundation Models を使わないため、Apple Intelligence 対応端末でなくとも単独では動作するが、constitution Principle IV により本プロジェクト全体の対象端末は Apple Intelligence 対応端末に絞られている。
- **共有 payload に含まれるメタデータ**: Apple 標準仕様により、ブラウザアプリ (Safari、Chrome iOS 等) は共有操作時にページタイトルを共有 payload に含めて渡す。本 spec はこれに依拠する。Title が含まれない場合は URL のホスト名にフォールバック。
- **単一ユーザー / 単一端末**: iCloud / CloudKit 同期は本 spec のスコープ外。データは端末内ローカルストアにのみ保存される。
- **重複検出 (URL 完全一致)**: 同一 URL を 2 度共有された場合、新規レコードを作らず「既に保存済みです」と通知する (Pocket 方式の savedAt bump は採用しない)。比較は文字列としての完全一致のみで、URL 正規化 (末尾スラッシュ・クエリパラメータ・トラッキング ID の除去等) は行わない。正規化を伴う重複判定や、重複時のマージ機能は将来 spec で扱う。
- **共有受け取り UI は最小**: ユーザー側でのタイトル編集や説明追記等の編集 UI は提供しない。共有 → 自動保存 → 自動クローズ の最小フロー (Principle II + Principle V)。編集機能を求める要望があれば将来 spec で扱う。
- **削除はネイティブ iOS スワイプ**: 削除確認ダイアログや undo スナックバーは MVP では持たない (Apple HIG 準拠)。
- **ネットワーク機能なし**: 本 spec はネットワークアクセスを一切持たないため、Principle I の「外部送信が発生する場合は spec.md で根拠記録」要件は本 spec では発生しない。

## Out of Scope

本 spec では以下を **明示的に扱わない**。すべて将来 spec で扱う想定。

- **本文抽出 / meta description / OG image 取得**: 次の spec「本文取得・メタデータエンリッチメント」で 1 回の HTTP fetch にまとめる。
- **要約 / カテゴリ分類** (Apple Foundation Models 利用): MVP コア機能だが、本 spec の保存基盤が完成してから別 spec で取り組む。
- **Safari Web Extension** によるワンタップ保存: constitution Principle IV により将来 phase。
- **iOS Shortcut 自動化** (Chrome を開いたら自動送信等): constitution Principle IV により将来 phase。
- **iCloud / CloudKit 同期**、**複数端末対応**。
- **URL 正規化を伴う重複判定** (末尾スラッシュ・クエリパラメータ・トラッキング ID 等の除去) と **重複時のマージ機能**。MVP では URL 完全一致による重複拒否のみ実装。
- **タグ / お気に入り / 検索 / フィルタリング**。
- **記事のメモ追記 / 手動タイトル編集**。
- **エクスポート / 共有 (一覧から OPML / JSON 等で書き出し)**。
